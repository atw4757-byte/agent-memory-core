# AMB v2.0.1-preview results

## Headline

- The pre-registered ordinal prediction is **FALSIFIED**. Actual ordering
  is the reverse of what was predicted.
- Scores are **identical across all 3 seeds and all 4 noise rates** —
  a methodology gap, not a code bug. We publish it rather than hide it.
- `agent-memory-core` — the library this benchmark ships with — is the
  **worst-performing adapter** on this grid. Reported honestly per the
  pre-registration contract.

## Pre-registered prediction vs. observed

**Pre-registered (PREREGISTERED.md §5):**

```
agent-memory-core (tuned) > agent-memory-core (stock) ≈ langchain (tuned)
  > langchain (stock) > naive
```

**Observed (all 4 noise rates, 3/3 seeds):**

```
naive ≈ langchain (stock) > langchain (tuned) ≈ agent-memory-core (stock)
  ≈ agent-memory-core (tuned)
```

**Verdict: ORDINAL FALSIFIED.** The prediction is not merely off — it is
reversed. Per the pre-registration contract:

> *"If the published alpha numbers diverge from the predictions below,
> the divergence is a finding — not a bug, and not an excuse to revise
> the formula."*

We are not revising the formula. We are recording the miss.

## Why the prediction missed (diagnosis, not justification)

1. **Noise distractors are lexically off-topic.** `NOISE_TEMPLATES`
   contains phrases like "asked for a joke" and "restaurant
   recommendations nearby." They share zero content words with
   scenario queries. Word-intersection retrieval (naive) and many
   embedding models treat them as uninteresting — so they never
   displace correct chunks from the top-5. This is why **noise rate
   has no measurable effect at any level tested.**

2. **Substring-containment matching over top-5 chunks favors
   recall-maximizing strategies.** `naive-append-only` returns its 5
   best-overlap chunks joined by `|`; if any one contains the expected
   phrase, the query scores a hit. Consolidation (the supposed
   differentiator for "tuned" modes) can only hurt this by removing
   matching chunks. `langchain-buffer tuned` drops chunks older than
   30 days; on scenarios whose answers live at days 0–4, that is
   exactly what happens (Q@90 collapses to 0.341).

3. **`agent-memory-core` underperforms because of the embedding
   retriever's recall at `n=8`.** The adapter calls
   `self._store.search(question, n=8)` and joins top-5 texts. Where
   word-overlap retrieves the exact lexical match, embedding retrieval
   returns semantically-adjacent chunks that do not contain the
   expected substring. This is a **retrieval-fit gap**, not evidence
   that the library is broken — but it is evidence that the adapter
   wrapper is not showing the library's strengths on this grid.

4. **Stock vs. tuned indistinguishable for `agent-memory-core`**
   because the wrapper's `consolidate()` calls
   `self._store.consolidate()` which is not exposed on `MemoryStore`
   in the installed v1.x API. The tuned flag is silently a no-op.
   This is wired correctly — the adapter catches `AttributeError` and
   logs — but it means the stock/tuned comparison for this adapter is
   currently uninformative.

## What is and isn't a valid claim

- **Valid:** The harness, simulator, metrics, and scoring pipeline are
  reproducible end-to-end on public + held-out scenarios.
- **Valid:** The pre-registered ordinal is falsified by data collected
  under the pre-registered methodology.
- **NOT valid:** "naive-append-only is the best memory adapter." The
  grid does not differentiate on noise; the "winner" is simply the
  adapter that retains the ground truth longest.
- **NOT valid:** "`agent-memory-core` is worse than a naive list."
  The retrieval-fit gap plus the stock/tuned no-op means this grid
  doesn't test the features that make the library useful in production
  (persistence, metadata, working-memory).

## What needs fixing before a real leaderboard claim (v2.1)

These are known gaps, deferred deliberately rather than hidden:

1. **Lexically-loaded noise distractors.** Replace the NOISE_TEMPLATES
   constants with phrases that share one or two content words with
   real query keywords. Without this, noise rate is dead weight in the
   grid.
2. **Adversarial-query scenarios.** Queries whose answers require
   contradiction resolution across chunks (the test `tuned` modes
   should win), not just phrase lookup.
3. **`agent-memory-core` adapter v2:** call `WorkingMemory.recall()`
   for recency and `MemoryStore.search()` with `n≥20` + rerank before
   the top-5 join; expose a real `consolidate()` path. This is an
   adapter fix, not a scoring-formula change.
4. **Real-data validation (D12):** ConvAI or similar, following the
   same schema.
